## Primitives needed (build order)

Three primitives unlock all 5 Marcus apps + every future workflow app. Build order is dependency-based: each unlocks the next.

### Primitive 1: Workflow (single-stage, Tier 1)

**What it does:** Declare a queue + decisions + audit + notifications in 5-10 lines of Clear. The language generates: queue page with status filter, action buttons that record decisions, audit table, notification rows, status transitions.

**⚠️ KEYWORD COLLISION FOUND 2026-04-27 (iteration 1):** Clear ALREADY has a `workflow` keyword for AI-agent orchestration (Phases 85-90). Existing shape: `workflow 'Name' with state: + state has: + step 'X' with 'Agent Y'` — chains AI agents over shared state with conditional routing (`if state's X is Y: ... otherwise:`) and retry loops (`repeat until ..., max N times:`). See `parser.js:195` (`WORKFLOW: 'workflow'`), `parser.js:4083+` (parser code), `clear.test.js:21730+` (existing tests). This is a DIFFERENT primitive from human-approval queues — leave it alone.

**Resolution:** the new primitive uses **`queue for X:`** as the keyword. Reads naturally for what Marcus actually sees ("I have a queue of deals waiting for review"). Avoids overloading `workflow`. Three of Marcus's existing 5 apps already use the word "queue" in their name (Approval Queue, Internal Request Queue, etc.) — this matches user vocabulary.

**Canonical syntax (locked iteration 1):**

```clear
queue for deal:
  reviewer is 'CRO'
  actions: approve, reject, counter, awaiting customer
  notify customer on counter, awaiting customer
  notify rep on approve, reject
```

**Generated by the language:**
- `DealDecisions` table (audit trail) — auto
- `DealNotifications` table (outbound queue) — auto
- `GET /api/deals/pending` (filtered queue) — auto
- `PUT /api/deals/:id/approve` (etc. for each action) — auto
- Action buttons in detail panel that fire the URLs — auto
- "Decision History" + "Notification Queue" tables in the UI — auto
- Status transitions (`status` field updates) — auto

**Scope:** single-stage only. NO multi-stage, NO board view, NO settings page. Defer those until customer evidence demands.

**Earned by:** all 5 Marcus apps. Each shrinks from ~150 lines of hand-rolled Clear to ~40 lines using this primitive.

**Per-primitive plan:** `plans/plan-queue-primitive-tier1-04-27-2026.md` (to write next iteration).

### Primitive 2: Triggered email + workflow email queue

**What it does:** Two pieces, both required:

(a) **Triggered email — `when X status changes to Y: send email`** — a "when" trigger that fires email on workflow events, not just user clicks.

```clear
when deal's status changes to 'awaiting':
  send email to deal's customer_email:
    subject is 'We countered your offer'
    body is counter_body
    track replies as deal activity
```

(b) **Durable email queue + provider abstraction** — every queued email gets a row the app can show. `provider is 'agentmail'` (default for reply-aware) or `provider is 'sendgrid'` (one-way). Compiler emits queue insert + background-worker send + retry logic.

**Generated by the language:**
- `WorkflowEmailQueue` table — auto (one row per queued send: to, subject, body, provider, status, error, attempts, sent_at, replied_at)
- `WorkflowEmailEvents` table — auto (provider callbacks: delivered, bounced, replied, failed)
- "Email log" view per parent record — auto
- Status changes on the parent (e.g. deal moves out of awaiting) when reply lands — auto

**Provider decision (from research):**
- **AgentMail** — default for reply-aware workflows (Marcus's deal desk needs customer replies to advance state)
- **SendGrid** — one-way transactional fallback (already exists as `send email via sendgrid:`)
- **Resend** — future option if developer-experience-first matters more (3K/mo free, React Email lib)

**Earned by:** Deal Desk + Approval Queue + Onboarding Tracker — all need notify-and-listen patterns.

**Per-primitive plan:** `plans/plan-triggered-email-04-27-2026.md` (to write).

**Safety:** real provider sends ONLY behind an explicit `enable live email delivery via X` directive + env-var keys. Default is queue-only (no real email in tests / preview / dev).

### Primitive 3: CSV export

**What it does:** Auto-add a "Download as spreadsheet" button to any workflow queue page. Per [Russell's GTM doc](../GTM.md) "What to build for the first customer" — "Export if they currently use spreadsheets."

**Canonical syntax (proposed):**

```clear
workflow for deal:
  ...
  export queue as csv  # one line, generates download button + CSV stream URL
```

Or even simpler: AUTO-include an export button on every workflow queue by default, with `no export` as the opt-out.

**Generated:**

- Download button in queue page header — auto
- CSV escaping + auth check — auto

**Earned by:** all 5 apps. Marcus moves FROM spreadsheets, but spreadsheets stay in his workflow for board reporting / handoffs / external sharing.

## Deferred (for after first customer evidence)

- **Multi-stage workflows** (Tier 2) — wait until expense tracker or another multi-stage app exists to validate the syntax shape
- **Settings page primitive** — Marcus's workflow changes quarterly, not daily; not MVP per GTM doc
- **Trello-style board view** — drag-to-advance breaks the audit story for approval flows
- **Connector platforms — TWO LANES, DIFFERENT DEFAULTS:**
  - **Lane A: AI assistant picks tool at runtime** → **Composio free tier** (20K calls/mo free; MCP-native; matches Clear's existing AI-assistant pattern)
  - **Lane B: Admin panel directly talks to HubSpot / Salesforce / Stripe / etc.** → **Nango free tier** (handles the OAuth permission dance + token refresh; ~$1/active-connection/month if scale; open-source self-host fallback)
  - **Lane C: One-way notifications to Slack / Discord / Teams** → **Direct webhook** — no platform needed. Likely a small Clear primitive (`send slack message to <webhook_url>`) when the first use case hits.
  - **Today:** build NONE of these. Marcus's 5 apps don't need external services yet.
  - Full pricing comparison + lane rationale in `snapshots/marcus-market-evidence-04-27-2026.md`.
- **Reply-parsing intelligence** — "did the customer accept or counter?" inference from email body — needs LLM integration; defer to after the basic queue + reply tracking lands
