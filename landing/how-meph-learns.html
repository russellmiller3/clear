The Flywheel

How we make Meph smarter without retraining

Most AI code assistants forget everything between conversations. This page explains the architecture we built to fix that for Clear — without changing the AI model itself — and an honest read on what we're seeing today.

The problem with AI coding tools right now

You open Cursor. Or Lovable. Or Bolt. You describe what you want. The AI generates code. It has a bug. You fix the bug by hand. You close the tab.

Tomorrow, you open it again. It makes the same bug.

Every AI coding assistant today has this problem. The AI doesn't remember your previous session. It doesn't remember anyone else's sessions either. Every conversation starts from a blank memory. Every user pays for the same mistakes, forever.

Imagine a contractor who rebuilds your kitchen every week and forgets which outlets are which each time. That's current AI code generation.

This isn't the contractor's fault — it's how the tools are architected. The AI gets a blank slate for every conversation. To fix a bug permanently, you'd have to retrain the AI itself, which costs millions of dollars and requires access most companies (including ours) don't have.

Three ways to make an AI smarter

When someone says "make the AI better," they usually mean one of three things:

1. Train a bigger model

What OpenAI / Anthropic / Google do. Billions of dollars. Years of compute. You get GPT-5 instead of GPT-4.

Not available to us. You can't train Claude — only Anthropic can.

2. Fine-tune an existing model

Teach an existing model your specific domain. Cheaper — thousands of dollars, weeks of work. Requires API access that Anthropic doesn't offer for Claude.

Also not available. Locked behind a door we don't have keys to.

3. Improve what the AI sees

Don't touch the model. Change the information you hand it. Better prompts, better examples, better context before it starts writing.

This is what we can do. And it turns out it's enough.

Option 3 sounds less impressive, and for most people's intuition about AI it is — "just give it better prompts" doesn't feel like the same magic as training a new model. But it has a secret property: it compounds. Every conversation teaches the system something it can feed into the next one.

Our version: the flywheel

Here's what Meph (Clear's AI assistant) does now, step by step:

  1. You ask Meph to build something. Say, an approval queue for expense requests.
  2. Meph writes code, compiles it. It fails with an error.
  3. Before Meph tries again, the system looks up 3 past sessions where someone else (or you, earlier) hit the same kind of error and eventually fixed it. It shows Meph those working examples.
  4. Meph patterns-matches off them, adapts to your situation, and tries again. This time it works.
  5. That session — error, fix, outcome — gets written to a database. Every future session benefits from it.

The crucial thing: Claude doesn't change. It's the same model. What changes is the information we hand it before it starts thinking.

Analogy. Claude is like a chef. A very good one. We're not trying to make the chef smarter. We're handing them a recipe book that keeps getting better — filled with dishes that worked, with notes on what went wrong the first time, with fixes that future chefs can reference.

How this relates to "real ML / RL"

If you've read anything about AI in the last few years, you've probably heard three phrases thrown around:

Machine Learning (ML)

The broad idea: computers learning from data instead of being explicitly programmed. Our flywheel is ML — we're learning which code fixes work for which errors. But we're not training a new neural network. We're training a small ranking model that picks the most relevant past examples to show Meph.

The model we train is called XGBoost. It's not a language model. It's a decision-tree ensemble — boring, fast, interpretable, about 22,000 parameters. GPT-4 has a trillion. Ours is five orders of magnitude smaller. It doesn't need to be big, because it's only doing one job: "given this situation, which past example is most relevant?"

Reinforcement Learning (RL)

Reinforcement learning is how AIs learn to play chess or control robots: try something, get a reward or punishment, update the strategy. What we're doing is RL-adjacent — we have "rewards" (tests passing is good, errors are bad), and we update which examples get recommended.

The difference: classical RL updates a policy network through backpropagation. We don't. We store past experiences in a database and retrieve the relevant ones. It's closer to how a human expert operates — "I've seen this before, here's what worked last time." Less elegant than pure RL, but it works without fine-tuning access.

Retrieval-Augmented Generation (RAG)

RAG is the name for this exact pattern when it's used with language models. Instead of training the model on new documents, you give it access to a database and let it look things up. Most company chatbots that answer questions about internal docs use RAG.

Our flywheel is a kind of RAG, but for code fixes instead of documents. The "documents" are labeled trajectories: "when you hit error X in app type Y, here's what fixed it."

Why this is interesting business-wise

If you're a user, the thing you notice is Meph getting better over weeks. Small things — it stops making a mistake it used to make; it understands your app patterns faster; fewer iterations before a feature works.

If you're competing with us, this is harder to copy than you'd think. The training data doesn't exist in the wild. Nobody else has "here are 10,000 cases where AI tried to build an approval queue and the errors it hit and how those errors got fixed in the specific language Clear." We accumulate that data as a side effect of users existing. Our competitors would need their own users using their own compiler to start collecting it.

Vercel did something similar with hosting data. Stripe did it with payment failure patterns. The compounding wins aren't visible in the product for months, but once they kick in they're very hard to catch up to.

Honest limitations

Three things to be honest about:

It doesn't work without users

The flywheel spins faster with real traffic. Synthetic data (practice tasks) is a shortcut to get started, but the real learning comes from humans using the product for real work. Until that happens, we're bootstrapping.

The Claude model is the ceiling

However good Meph gets through retrieval, its ultimate ceiling is Claude itself. If Claude can't reason through a pattern no matter how many examples you show it, we're stuck. Fine-tuning would break that ceiling. We can't do it yet.

Bad data poisons the well

If the database fills up with wrong fixes that happened to pass superficial tests, the ranking model will confidently recommend bad patterns. We've built guardrails — only fully-passing sessions become training examples — but it's something to watch.

Hint delivery is the bottleneck right now

The mechanism works in principle: when a hint reaches Meph, he uses it about 70% of the time, and almost three-quarters of those uses are rated helpful. The catch is hints don't reach him very often. Today most rows in the database log no hint at all — the retriever's archetype filter is concentrated in a few app shapes (api_service, queue_workflow, routing_engine), and unfamiliar shapes slip through with nothing matched. We don't yet have positive lift evidence in a controlled A/B comparison. We believe the mechanism — every signal we have on individual hint quality is positive — but the across-the-board "Meph gets noticeably better as you use it" claim is still being earned, not measured. If you're considering Clear partly because of this flywheel, ask us where the measurement is at the moment.

The bigger story

For twenty years, the way you made software better was to hire better engineers. For the last three years, AI started doing a lot of the work. What we're building is the third chapter: the software that gets better on its own as it gets used.

Clear is a compiler that writes itself smarter. Meph is an AI that learns without needing to be retrained. The flywheel is the engine connecting them. Every user's session makes every other user's next session a little easier.

That's the whole thesis in one paragraph.
