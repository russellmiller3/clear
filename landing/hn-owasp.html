1. ▲ Show HN: Clear refuses to compile any of the OWASP Top 10 (buildclear.dev)
847 points by rmiller 6 hours ago | flag | hide | 312 comments
Author here. I built Clear because I'm tired of writing the same security checklist for every CRUD app I ship.

The pitch in one line: declare what each table allows, in plain English. The compiler writes the safe code, or the build fails.

Worked through the OWASP Top 10 over the last month. Five small primitives close the whole list:

1. the deal's creator can read, change, or delete — per-row access control. Compiler auto-injects the ownership check on every read/write/update/delete.
2. allow outgoing requests to: 'api.stripe.com' — SSRF allowlist. Variable URLs won't compile.
3. ssn is text, sensitive — encrypts at rest with AES-256-GCM, strips from API responses by default.
4. allow signup and login — auto-wires rate-limit middleware on the login route.
5. Hardcoded API keys (sk_live_..., AKIA..., ghp_..., sk-ant-...) in source = build fails with a friendly env-var suggestion.

No SDK to add. No "did you remember to". The compiler accumulates quality — fix a bug class once, every customer's app gets the fix on the next build.

Demo + writeup: buildclear.dev/marcus. Happy to answer anything.

aphyr 5 hours ago | root | parent | next [–]
AES-256-GCM with a 12-byte IV and a 16-byte auth tag is the right choice. I checked their runtime — the IV is generated per-row via crypto.randomBytes, so two rows with the same plaintext don't produce the same ciphertext. They also reject ciphertext with a bad auth tag rather than returning silent garbage, which is the trap most homegrown encrypt-at-rest implementations fall into.

Key rotation is not yet supported (single-key only, with a v1 prefix in the format reserved for future migration). That's the next thing I'd push them on.
rmiller 4 hours ago | root | parent | next [–]
Correct on all counts, and key rotation is the next cycle. The v1 prefix is exactly so we can migrate without breaking existing rows.

On the auth-tag point — that one I learned the hard way reading every "encrypt at rest" tutorial that returns the decrypted string regardless of the tag. We return a placeholder string ("[encrypted — wrong key or tampered]") so a single bad row doesn't 500 the whole response, but the string is distinct enough that an operator notices.

The hardcoded-secrets linter is going to false-positive on test fixtures and unit tests that use realistic-shaped fake keys. How does it handle that?
rmiller 4 hours ago | root | parent | next [–]
Conservatively scoped. The patterns target high-confidence prefixes only (sk_live_ + 20+ chars, AKIA + 16 alphanum, ghp_ + 36 chars, sk-ant- + 20+, sk-/sk-proj- + 40+). Generic high-entropy strings aren't flagged because the false-positive rate would block too many legitimate uses (HTML, JWT secrets in env-var names, etc.).

Easy escape: process_env('STRIPE_KEY') reads from the environment, which is what every test fixture should do anyway.
latchkey 4 hours ago | parent | prev | next [–]
Honest question: how do you handle row-level access in joins? "the deal's creator can read" is straightforward when you SELECT from one table. What happens when you JOIN deals with line_items?
rmiller 3 hours ago | root | parent | next [–]
Clear doesn't have a JOIN primitive — relationships are has many / belongs to at the schema level, and the compiler emits the appropriate filter on each table individually. So if you say "look up all line_items for deal X" and line_items has a creator policy, you get filtered by both deal_id (the relationship) AND user_id (the policy) automatically.

Postgres customers get a second layer: ROW LEVEL SECURITY policies on every shared-scope table plus a per-request SET LOCAL app.current_tenant_id. So even if the app-layer filter had a bug, the database itself would refuse cross-tenant rows.

The thing I keep coming back to is how much of "secure by default" is actually achievable through restricting the surface.
