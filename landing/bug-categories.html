Compiler-Enforced Quality

45 Bug Categories. Eliminated at Compile Time.

Other tools generate code and hope it works. Clear's compiler structurally prevents entire classes of bugs — security vulnerabilities, correctness errors, and runtime failures — before a single line of output runs.

Security 7

  • SQL injection — DB uses filter objects, never string concatenation
  • Stored XSS — all output auto-escaped, DB writes sanitized
  • Missing auth on DELETE/PUT — compiler refuses to compile
  • IDOR — compiler blocks unfiltered queries on user-owned tables
  • Privilege escalation — warns when POST accepts owner_id from client
  • Plaintext passwords — PBKDF2 with salt built into auth runtime
  • Token forgery — HMAC-SHA256 signature verification automatic

Correctness 8

  • Forward references — variables must be defined before use
  • Type mixing — string + number flagged at compile time
  • URL mismatch — compiler verifies frontend fetch URLs match backend endpoints
  • Duplicate endpoints — compiler error on same method + path
  • Missing validation — auto-generated from data model
  • Wrong HTTP status codes — compiler sets 201 for POST, 400 for errors
  • Undefined functions — caught at compile time
  • Schema field mismatch — validation rules cross-checked against DB fields

Runtime Safety 8

  • try/catch on every endpoint — automatic, no silent 500s
  • Null/undefined guards — utility functions check before access
  • localStorage errors — wrapped in try/catch (Safari private mode)
  • CORS configuration — auto-applied when specified
  • Middleware ordering — compiler ensures logging, CORS, then auth
  • Structured error responses — JSON errors, not stack traces
  • Request body validation — checks req.body exists before access
  • Graceful shutdown — built into every server output

Plus 22 more across input validation, auth/crypto, type coercion, and data persistence.

See It In Action

Eight Lines. Every Bug Handled.

This is a real Clear endpoint. Read every line out loud — it means exactly what it says.

when user calls PUT /api/invoices/:id/send:
  requires auth
  define invoice as: look up the Invoice with this id
  guard invoice is not nothing or "Invoice not found"
  guard invoice's status is "draft" or "Only drafts can be sent"
  invoice's status is "sent"
  save invoice to Invoices
  send back "sent" with success message

From these 8 lines, the compiler generates:

  • ✓ Express route handler with async (req, res)
  • ✓ JWT auth middleware — rejects unauthenticated requests
  • ✓ try/catch wrapping the entire handler
  • ✓ Guard clauses with proper 400/404 status codes
  • ✓ XSS-safe response — all output escaped
  • ✓ Correct HTTP 200 response with JSON body

A developer writing this by hand would need 30-40 lines of Express.js and would likely forget at least one of those protections.

Why This Matters

Other AI Builders Skip All of This

Lovable, Bolt, and v0 generate code from scratch every time. They produce working apps — but with none of these protections. No compile-time security checks. No auto-generated validation. No structural guarantees.

When Lovable generates a DELETE endpoint, it does not check for auth. When Bolt generates a form handler, it does not validate input types. When v0 generates a data table, it does not escape HTML. These are not edge cases — they are the default.

Clear's compiler catches these at build time. Not as warnings you can ignore — as errors that block compilation until you fix them.
